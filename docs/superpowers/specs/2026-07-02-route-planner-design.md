# 目標貓最短抽卡路線規劃 — 設計

日期:2026-07-02

## 背景與問題

比較頁已能並排看到各卡池同抽數的 A/B 軌內容。常見需求是「我想要這幾隻貓,從 1A 出發
要怎麼抽最省」。因為所有卡池共用同一條種子序列,且會因**重複稀有換軌**改變你能經過的
位置(A↔B 軌),徒手規劃很難。本功能讓使用者挑目標貓,工具算出從 1A 收齊目標的路線。

## 目標

- 使用者挑「想抽到的貓」(點格 / 選貓名),工具從 **1A** 出發規劃收齊全部目標的路線。
- 正確處理**換軌**(本次僅重複稀有;無保證 Uber / GU,先不做)。
- 抽數與特殊券(白金券/傳說券)之間會有取捨,故輸出 **Pareto 前緣**多方案讓使用者挑。

## 非目標

- 不自算種子;忠實沿用 godfat / parser 已解析的資料。
- 不改 parser / godfat / concurrency / cache。
- 本次不處理保證 Uber(GU 11 連 / step-up)換軌。筆記 07 的「保證欄乾淨起點」實測疑慮
  本次不觸及。
- 不做跨頁面 / 重抓資料;完全複用比較頁已載入的資料。

## 核心機制模型(皆可由已解析資料計算,零種子運算)

### 位置圖(2026-07-03 更正:B→A 落點為 n+2)
- 起點 = **1A**。正常抽 → `(n+1, 同軌)`。
- 重複稀有換軌落點**依方向不同**(A/B 軌是同一種子序列的交錯編號,nB 夾在 nA 與
  (n+1)A 之間;重抽多耗一顆種子):
  - A 軌觸發 → `(n+1, B)`
  - B 軌觸發 → `(n+2, A)`(**跳過位置號 n+1,省一抽**)
  - 落點優先採 godfat 已標的 `dupe.to`(可帶 R 字尾=抵達時仍處重複狀態);未標時按上述規則。
- 推論修正:**總抽數 = 實際抽的步數 k**,因 B→A 換軌跳號,**可能少於最遠收集位置**。

### 重複稀有觸發(跨卡池、以貓名判定)
- 在位置 `n`、目前軌 `track`、選卡池 `b` 抽:令 `nat = cell(b, n, track)`。
- **觸發換軌 iff** `nat.rarity === 'rare'` 且 `nat.name === 上一抽實際取得的貓名`。
- 連續兩隻 SR / Uber / Legend 不觸發(僅 `rare` 判定)。
- 起點特例:1A 是否觸發取決於取得種子時的 `last`(上一抽)。本模型的「上一抽貓」初值 =
  該 `last` 對應的貓名(若比較頁 URL 帶 `last`;無則視為無、1A 不因撞 last 換軌)。

### 換軌落點與重抽所得貓
- **落點**:重複稀有於位置 `n` 觸發 → 落 `(n+1, 對軌)`(確定,不需種子運算)。
- **重抽所得貓**:
  - 若 godfat 已標該格 `dupe`(parser 的 `cell.dupe` 存在)→ 用其 `dupe.name` / `dupe.rarity`
    (該重抽結果僅取決於「位置種子 + 被排除的天然貓」,與如何觸發無關,故跨卡池仍有效)。
  - 否則(godfat 未標,例:跨卡池才觸發的重複)→ 標記為**不確定**:此步僅用於換軌,
    **不得**當作收集到某目標;且其「取得貓名」設為特殊值,保守假設**不**造成下一抽連鎖重複。

### 特殊券卡池(白金 / 黑金)
- 依比較頁 `banner-names.js` 的簡稱判定(`白金` / `黑金`)。
- **必不換軌**(白金/傳說票不換軌);依**貓名**當一般格對待(採用使用者「依名字為主」的選擇;
  註:godfat 白金/黑金欄的貓名是共用種子天然貓、與實際票抽必為超激的結果不同,此為已知取捨)。
- 每次於白金抽 → 消耗 1 **白金券**;於黑金抽 → 消耗 1 **傳說券**。

### 目標定義
- **點格目標**:某一個 `(banner, pos)` 格,單一可接受格。
- **貓名目標**:某貓名,其在所有已載入卡池 / 位置的**任一 occurrence** 皆可滿足。
- 內部統一表示:每個目標 = 一組「可接受格集合」`{(banner, pos)}`;點格 = 單元素集合。

## 成本與最佳化

- **每位置至多收 1 個目標**(一次單抽)。若兩個目標的可接受格在同一位置且無其他 occurrence
  可替代 → **互斥**。
- 成本三軸(皆最小化):
  1. **總抽數**(實際步數 k;B→A 換軌跳號時會少於最遠收集位置)
  2. **白金券數**
  3. **傳說券數**
- 並列 tie-break:**最少換軌次數**。
- 輸出 **Pareto 前緣**:互不被支配的方案集合(A 支配 B iff A 三軸皆 ≤ B 且至少一軸 <)。
  同一 Pareto 點若有多路線,取換軌最少者為代表。

## 演算法

- **多目標標籤法(label-setting)前向搜尋**。因位置嚴格遞增,狀態圖為 DAG,依位置推進。
- **狀態** `key = (pos, track, collectedMask, lastCatName)`。
  - `collectedMask`:已收集目標的 bitmask(目標數 ≤ 12)。
  - `lastCatName`:上一抽實得貓名(用於判斷下一抽是否重複稀有);不確定重抽以特殊哨兵值表示。
- **標籤** = 該狀態下的一個 Pareto 成本向量 `(pulls, plat, legend, switches)` + 回溯指標。
  每狀態保留其 Pareto 標籤集(丟棄被支配者)。
- **轉移**:於 `(pos, track)`,對每個可見卡池 `b`:
  - 判定是否觸發重複稀有 → 決定下一 `track` 與 `lastCatName`、`switches` 是否 +1。
  - `pulls += 1`;若 `b` 為白金/黑金則對應券 +1。
  - 若此格屬某未收集目標的可接受格且非「不確定重抽」→ 於下一狀態把該目標位元設為 1。
    目標可帶 `name`(期望貓名):此時**實得貓名(`gotName`)須與 `name` 相符**才計收集——
    dupe 觸發與否會改變實得貓,僅憑位置會把「拿到別隻」誤計。同格若滿足多個目標
    (名字皆與實得相符,即同名目標)可同時計入。
- **邊界**:`pos` 上限 = 已載入 `count`;目標數上限 12(超過則於 UI 擋下並提示)。
- **終止 / 取解**:所有 `collectedMask == full` 的標籤即候選解;跨所有終點狀態取三軸 Pareto 前緣。
- **不可行**:若無任何標籤達成 full mask → 對「可行子集」求最大可收集集合(依相同搜尋,取
  collectedMask popcount 最大且 Pareto 最省者),並回報未能納入的目標與原因(互斥 / 不可達)。

### 模組介面(`extension/lib/route.js`,純函式)

```
planRoutes({ merged, targets, banners, options }) -> {
  feasible: boolean,
  plans: Plan[],              // Pareto 前緣;不可行時為「最大可行子集」的方案
  unreachable: TargetRef[],   // 未能全數納入時,列出未納入目標與原因
}

Plan = {
  cost: { pulls, plat, legend, switches },
  steps: Step[],             // 逐步指令
  collected: TargetRef[],
}
Step = {
  k, pos, track, bannerId,
  gotName, gotRarity,
  switched: boolean,         // 此抽是否換軌
  uncertain: boolean,        // 重抽結果不確定
  ticket: 'plat' | 'legend' | null,
  collectedTarget: TargetRef | null,
}
```
- `targets`:`[{ id, kind:'cell'|'cat', accept: Set<'${bannerId}|${pos}'> }]`。
- `banners`:可見卡池順序與其 `id` / `short`(判特殊券)。
- `options`:`{ maxPos, maxTargets }`(帶預設)。

## UI(融入比較頁 `extension/view/`)

- **選目標**
  - 點資料格切換 target 標記(醒目邊框 + 角標);再點取消。
  - Find 面板加「把目前命中全加為目標」按鈕(貓名目標:把該貓所有 occurrence 併為一個目標)。
  - 目標以可移除**晶片**列出(工具列下方,類似卡池開關列)。
- **規劃路線**
  - 「規劃路線」按鈕 → 呼叫 `planRoutes` → 底部抽屜(沿用 Find 抽屜樣式與收合/拖曳)。
  - 抽屜上半:Pareto 方案清單,每列顯示 `抽數 / 白金券 / 傳說券 / 換軌數`,可點選。
  - 選一方案 → 主表**依序高亮路徑**:經過格標步序號、target 格強調、換軌點特別標、
    用券格標券別;不確定重抽步標註。
  - 不可行時:抽屜頂部提示未納入目標與原因,清單改列「最大可行子集」方案。
- 白金/黑金:載入哪些卡池就納入哪些(無額外開關)。

## 測試(`test/route.test.js`,純函式、合成模型、不碰網路)

以手工建構的 `merged`(`{ numbers, byPos }`)測:
1. **直線收集**:目標都在 A 軌,回傳單一直走方案、抽數 = 最遠目標位置、0 換軌、0 券。
2. **需換軌收 B 軌**:某目標在 B 軌,路線於適當位置以重複稀有換軌後收齊。
3. **白金券 trade**:同貓在「近白金格」與「遠一般格」各一,Pareto 前緣同時含
   `少抽多券` 與 `多抽少券` 兩方案。
4. **同位置互斥不可行**:兩目標唯一可接受格落同位置 → `feasible:false`,`unreachable` 列出。
5. **不確定重抽標記**:跨卡池才觸發、godfat 未標 dupe 的換軌步 → `uncertain:true` 且不計為收集。
6. **tie-break**:兩方案三軸相同、換軌數不同 → 取換軌少者為代表。

## 風險 / 取捨

- 狀態含 `lastCatName` 使搜尋較重;以 `maxPos`(= 載入 count)與 `maxTargets`(12)控制。
  若實測偏慢,plan 階段可再簡化(例:限制換軌次數上限、或先只用 godfat 已標的 dupe 邊)。
- 白金/黑金「貓名 ≠ 實際票抽貓」為已知取捨(依使用者選擇以名字為主),結果會標明券別供判讀。
- 不確定重抽:僅用於換軌、不當收集,並保守假設不連鎖;真實遊戲若不同,屬既有 godfat 資料限制。

### 1. 卡池時間(availability window)約束

路線目前忽略卡池的活動時間,可能產出時間上不可行的抽序。需加入:

- 每個卡池有活動時間窗(比較頁已有 `banner.date` = `start~end`,來自 `bannerDateRange`)。
- **只能在卡池活動期間抽該卡池**。
- **時間單向前進**:抽卡在真實時間上依抽序推進(可往未來時間旅行,見 `docs/seed-tracking/04-進階技巧.md`,
  但**不能先抽較晚(較後)開始的卡池、再回頭抽已結束的舊卡池**)。
- 形式化:沿遞增抽序所選的卡池,需存在一組非遞減時間戳 `t_1 ≤ t_2 ≤ …`,使每抽的卡池在該時間點皆
  在活動窗內。等同區間排程可行性。
- 實作方向:搜尋狀態加入「目前時間下界」;卡池 `b` 於某抽可用 iff `b.end ≥ 目前時間下界`,選它後把時間
  下界更新為 `max(目前下界, b.start)`。需先把 `banner.date` 的字串解析成可比較的日期(注意「進行中/無結束」
  與跨年格式)。時間不可行的目標 occurrence 應排除,並在 `unreachable` 以新原因(如 `time-conflict`)回報。

### 2. 白金 / 黑金 券別偏好(白金優先)

- 若**白金與黑金在同一位置給相同結果**(同一隻目標貓),**優先用白金**——白金券比傳說券便宜。
- 目前 Pareto 以 (pulls, plat, legend) 三軸並列,兩者互不支配。需加入偏好:結果等價時傾向消耗白金而非傳說
  (例:tie-break 時 legend 權重高於 plat,或在等價 occurrence 中預設選白金卡池)。
